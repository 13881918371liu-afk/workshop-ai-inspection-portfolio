# 车间 AI 巡检系统：第一版最小闭环

## 目标

先不做大系统，先跑通一条真实数据链路：

员工/主任提交问题 → AI 整理 → 飞书多维表格入库 → 老板/管理层在飞书看今日问题和闭环状态。

## 第一版边界

- 员工端：优先做微信小程序形态，扫码进入，尽量少按钮。
- 老板端：先用飞书看板，不急着做独立 App。
- 云端：飞书多维表格作为第一版数据底座。
- AI：只做口语整理、异常分类、标准名映射、摘要生成。
- 技术突破：先预留提醒入口，暂不接入生产技术研究数据。

## 员工端入口

第一版只保留 3 个入口：

1. 现场问题
   - 杯子、杯盖、空杯、污染、虫体异常、物料残留等。
2. 员工反馈
   - 员工临时拉主任反馈的问题。
3. 晨会纪要
   - 主任说一段话，AI 拆成昨日复盘、今日任务、质量重点、物料确认。

## 飞书多维表格建议

### 1. 原始上报表

记录所有从员工端进入的原始内容。

字段：

- 上报时间
- 上报人
- 车间
- 班组
- 来源入口
- 原始文本
- 图片/附件
- AI 处理状态
- 关联异常事件

### 2. 异常事件表

老板看板的核心表。

字段：

- 日期
- 车间
- 班组
- 问题类型
- 标准对象
- 问题描述
- 严重程度
- 责任人
- 状态
- 来源记录
- 处理结果
- 复查结果

状态建议：

- 待主任确认
- 已确认
- 待处理
- 处理中
- 已处理
- 已复查
- 已关闭

### 3. 车间知识库表

把员工口语映射成标准名。

字段：

- 口语/俗称
- 标准名称
- 所属车间
- 对象类型
- 备注

例子：

- 杯子 → 4 厘米布丁杯
- 杯盖 → 对应规格杯盖
- 空杯 → 杯体/杯盖密封风险或虫体缺失风险

### 4. 晨会纪要表

承接主任每天口述内容。

字段：

- 日期
- 车间
- 主持人
- 记录人
- 昨日复盘
- 今日生产任务
- 工艺与质量重点
- 物料确认
- 补充项
- 员工反馈汇总

## AI 处理节点

### 输入

员工或主任提交一句话：

> 旧杯旧盖盖不严，担心后面出现空杯，需要主任看一下。

### AI 输出

结构化结果：

- 车间：供虫车间
- 问题类型：物料/杯盖异常
- 标准对象：4 厘米布丁杯/杯盖
- 问题描述：旧杯旧盖密封不严，存在空杯风险
- 严重程度：一般
- 状态：待主任确认
- 是否生成异常事件：是

## 第一版真实回路

1. 员工扫码进入小程序。
2. 选择“现场问题”。
3. 说一句或手动输入。
4. 小程序把文本发到云函数/后端。
5. 后端调用 AI 整理成结构化 JSON。
6. 后端调用飞书 API 写入原始上报表和异常事件表。
7. 主任在飞书里确认。
8. 老板在飞书看板里看到今日问题和闭环状态。

## 现场需要拿到的资料

1. 飞书是否已经开通企业组织。
2. 谁是飞书管理员。
3. 是否允许创建飞书自建应用。
4. 是否允许开放多维表格 API 权限。
5. 五个车间人员名单和角色。
6. 每个车间最常见的 20 个问题。
7. 常用口语和标准名称对照。
8. 谁负责确认问题，谁负责处理问题。
9. 老板想看的飞书看板字段。

## 第一周建议

第一周不要追求完整语音识别和复杂 AI。

建议顺序：

1. 先搭飞书多维表格。
2. 用飞书表单或临时网页表单模拟员工入口。
3. 人工复制 20 条真实问题，测试 AI 能否分类。
4. 跑通异常事件表和老板看板。
5. 再决定微信小程序是否正式开发。

## 技术选择

第一版推荐：

- 员工端：微信小程序或临时 H5 表单
- 中间层：轻量后端/云函数
- 数据层：飞书多维表格
- AI：外部模型 API
- 老板端：飞书看板 + AI 问答入口

## 当前判断

最稳的落地方式不是一上来做完整 App，而是先让数据闭环真实跑起来。

如果飞书表格能承接数据和看板，老板端暂时不需要独立软件；员工端如果要低门槛使用，微信小程序比单独 App 更合适。
