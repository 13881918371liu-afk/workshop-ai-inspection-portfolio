# 车间 AI 巡检系统：样表信息提取与实现思路

## 1. 样表 A：晨会纪要/每日工作安排表

### 1.1 表格类型

这是一个“车间每日晨会 + 昨日复盘 + 今日安排 + 员工反馈”的综合记录表。

它适合做成语音/文字输入后由 AI 自动整理的表，因为大部分内容是自然语言描述，不是精确勾选。

### 1.2 表头字段

| 字段 | 当前样表内容 | 系统字段建议 |
| --- | --- | --- |
| 表名 | 生产车间班组每日工作安排与总结·晨会纪要 | 模板名称 |
| 车间 | 勾选：供虫车间 | workshop |
| 日期 | 2026 年 7 月 6 日 | date |
| 会议主持 | 纪丽 | host |
| 记录人 | 纪丽 | recorder |
| 缺席人员 | 空 | absent_people |
| 会议纪要 | 空 | meeting_notes |

车间选项包括：

- 前处理
- 供虫车间
- 侵染车间
- 巡检车间
- 成品车间

### 1.3 昨日工作复盘字段

原表栏位：

> 昨日工作复盘（产量/目标达成情况、抽检情况通报/问题与异常回顾）

当前样表内容可提取为：

1. 昨日（周六）领幼虫共 113 人，按标准流程完成 445/128 头。
2. 质量抽检情况：整体在区分漏虫、死虫，掌上有很大的进步。
3. 最近因用旧杯、旧盖会偶尔导致盖不严，出现空杯现象，所以要求员工把变型的杯盖挑选出来。

字段建议：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| yesterday_output_summary | 长文本 | 昨日产量/任务完成情况 |
| yesterday_quality_summary | 长文本 | 昨日质量抽检情况 |
| yesterday_abnormal_summary | 长文本 | 昨日问题和异常 |
| yesterday_raw_text | 长文本 | 原始语音/手填文字 |
| has_abnormal | 复选框 | 是否存在异常 |
| abnormal_tags | 多选 | 如空杯、杯盖变形、漏虫、死虫 |

### 1.4 今日工作安排字段

原表栏位：

> 今日工作安排：（今日生产任务、工艺与质量重点、物料确认）

当前样表内容可提取为：

1. 今日翻幼虫 135 人，按标准操作流程完成生产任务量。
2. 要求员工在用未清洗的杯体、杯盖时注意变型和坏掉的，发现了就及时更换，余在杯子里的饲料和废土及时清理后才能用。
3. 物料确认：旧杯、旧盖未清洗，有残留的废土、饲料、霉粉。

字段建议：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| today_task | 长文本 | 今日生产任务 |
| process_quality_focus | 长文本 | 工艺与质量重点 |
| material_confirmation | 长文本 | 物料确认 |
| today_raw_text | 长文本 | 原始语音/手填文字 |
| task_people_count | 数字 | 如 135 人 |
| material_status | 单选/多选 | 如旧杯未清洗、旧盖未清洗、有残留 |

### 1.5 员工反馈字段

当前样表内容：

> 员工使用旧杯、旧盖导致空杯能否不少于 10 元/杯处理。

该句可能存在手写识别不确定，实际含义可能是：

- 员工反馈旧杯旧盖导致空杯，建议按不少于 10 元/杯处理；
- 或询问空杯能否按 10 元/杯处罚/处理。

字段建议：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| feedback_raw_text | 长文本 | 员工原始反馈 |
| feedback_summary | 长文本 | AI 整理后的反馈 |
| feedback_type | 单选 | 质量问题/物料问题/处罚建议/流程建议/其他 |
| related_issue | 多选 | 空杯、旧杯旧盖、杯盖变形等 |
| need_followup | 复选框 | 是否需要管理层跟进 |

### 1.6 适合 AI 介入的地方

该表非常适合 AI 做四件事：

1. 把主任的一大段口述拆成固定字段：昨日复盘、今日任务、质量重点、物料确认、员工反馈。
2. 把模糊词补全成车间专用语言：杯子 -> 供虫车间杯体/杯盖；旧杯旧盖 -> 未清洗旧杯旧盖。
3. 自动识别异常：空杯、杯盖变形、旧杯旧盖未清洗、废土饲料残留。
4. 自动生成管理层摘要：今日异常只报“旧杯旧盖导致空杯风险、物料未清洗、有残留”。

## 2. 样表 B：生产过程巡查记录

### 2.1 表格类型

这是一个“过程巡检清单 + 抽样记录 + 合格率统计”的表。

它不适合全部语音化，因为中间有大量格子、样号、组别、勾选，人工勾选可能比逐条口述更快。

它适合做成“结构化表单 + 批量默认合格 + 异常单独录入”的模式。

### 2.2 表头字段

| 字段 | 当前样表内容 | 系统字段建议 |
| --- | --- | --- |
| 表名 | 生产过程巡查记录 | 模板名称 |
| 班组 | 浸染件/制盖杯 | team_or_line |
| 巡查日期 | 2026 年 7 月 7 日 | date |
| 巡查人 | 拉蓉 | inspector |
| 信息记录/反馈 | 空 | feedback |

### 2.3 巡查项目及标准

表格中可识别的巡查内容包括：

#### a. 测场内溶/测场物料

标准大意：

- 物料按码规范、齐整；
- 无安全风险；
- 不妨碍紧急逃生等应急通道畅通；
- 物料状态标识及质量符合使用要求；
- 标识清晰可识别；
- 内容齐整，无涂抹；
- 确认物料外观形态、大小适合；
- 无霉败变质；
- 随机抽查 3-5 框；
- 观察外观、大小、质地；
- 确认是否缺水分；
- 随机抽取查看标识。

这类项目适合“主任说一句现场物料合格，系统自动全勾选”。若异常，只需说异常项和原因。

#### b. 操作过程

标准大意：

- 抽查针刺刻操作规范性；
- 以桌为组单位，随机抽查 5 组；
- 频率 1 次/2h，3 个样/次；
- 观察重点：进针深度、针刺部位、进液速度、安装排气、过强黏针；
- 合格打勾，不合格打叉；
- 抽查盖杯操作质量；
- 以桌为组单位，随机抽查 3 组，频率 1 次/2h，3 个样/次；
- 判别标准：饲料、土装填、次品虫的剔除，每 200 头不超过 5 头；合格打勾，不合格打叉。

这类项目细到样号 01-16、A/B/C/D/E 组、随机抽查记录。第一版建议保留人工录入或表格直接勾选，不强行语音化。

### 2.6 适合 AI 介入的地方

该表 AI 的价值不是替代所有勾选，而是做：

1. 默认合格批量勾选：如“现场物料今天合格”，系统自动给该分类下所有检查项打勾。
2. 异常口述归类：如“B 组 7-3 有病虫 1 个”，AI 自动写入 B 组、批次 7-3、异常类型病虫、数量 1。
3. 自动计算合格率：1200 个、1199 个合格 -> 99.91%。
4. 输出异常摘要：只展示“B 组 7-3 发现病虫 1 个”等问题，不展示所有正常勾选。

## 3. 从样表反推系统设计

### 3.1 第一版不要做成“万能 AI”

第一版应当做成“模板驱动的 AI 表格助手”：

- 每个车间先建自己的模板；
- 每个模板固定字段；
- AI 只负责把原始话术填入对应字段；
- 管理层看板只读取异常字段。

这样最容易落地，也最容易让车间主任接受。

### 3.2 飞书多维表格建议结构

建议先建 5 张基础表：

1. 车间表：车间名称、负责人、常用物料、默认模板。
2. 晨会纪要表：适配样表 A。
3. 巡查记录表：适配样表 B 的主记录。
4. 巡查项目明细表：适配样表 B 的细项和抽样结果。
5. 异常事件表：所有车间异常统一汇总。

### 3.3 异常事件表是老板看板核心

字段建议：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| date | 日期 | 异常日期 |
| workshop | 单选 | 车间 |
| source_table | 单选 | 晨会/巡查/员工反馈/其他 |
| issue_type | 多选 | 空杯、病虫、死虫、物料残留、杯盖变形等 |
| issue_summary | 长文本 | 异常摘要 |
| severity | 单选 | 轻微/一般/严重 |
| owner | 人员 | 负责人 |
| status | 单选 | 待处理/处理中/已处理/已复查 |
| evidence | 附件 | 图片/原始表格 |
| raw_text | 长文本 | 原始记录 |

### 3.4 车间知识库建议

第一版知识库可以很轻，不需要训练模型。

字段建议：

| 字段 | 类型 | 示例 |
| --- | --- | --- |
| workshop | 单选 | 供虫车间 |
| alias | 文本 | 杯子 |
| standard_name | 文本 | 4 厘米布丁杯/杯体/杯盖 |
| issue_alias | 文本 | 破了、坏了、发霉、盖不严 |
| issue_type | 单选 | 物料异常/质量异常 |
| normalized_output | 文本 | 供虫车间杯体/杯盖发生异常 |

## 4. 建议的第一期范围

第一期建议只做两个模板：

1. 供虫车间晨会纪要模板。
2. 浸染/制盖杯生产过程巡查模板。

先实现 4 个动作：

1. 主任输入一段文字或手机语音转文字。
2. AI 拆成固定字段。
3. 写入飞书多维表格。
4. 异常自动同步到老板看板。

不建议第一期做：

- 独立 App；
- 实时语音识别；
- 全厂五车间深度定制；
- 复杂权限系统；
- 自动识别手写图片。

## 5. 当前最需要补充的资料

为了继续推进，需要客户补充：

1. 五个车间各自的空白表格模板。
2. 每个车间最近 3 天的已填写样表，最好脱敏。
3. 车间主任每天最常说的 20 个问题/异常。
4. 老板最想看的 10 个异常指标。
5. 哪些内容必须留原表格式，哪些可以改成更好用的飞书视图。

## 6. 初步结论

样表 A 是 AI 价值最高的入口：自然语言多、结构固定、可以明显减少手写。

样表 B 是数据价值最高的入口：检查项细、样本数和合格率明确，适合形成异常看板，但不适合完全语音化。

系统第一版的关键不是“让 AI 全自动”，而是让 AI 帮主任少写字、帮老板少看废话、帮异常有记录可追溯。

## 7. 行业理解补充：为什么要这样设计表格

从公开资料看，蛹虫草/虫草类工厂化生产的核心不是简单“种植”，而是围绕菌种、虫体/培养基、接种或侵染、培养环境、污染控制、采收质量进行稳定复制。

对本项目有用的理解：

1. 虫草类产品的价值和虫草素、多糖等活性成分、外观品质、污染控制、批次稳定性有关。
2. 工厂每天真正要控的是过程变量：虫体状态、杯体杯盖、饲料/土、清洁程度、接种/侵染质量、温湿度、污染、死虫、病虫、空杯、抽样合格率。
3. 车间表格不能只当“工作日志”，应该当“生产过程数据库”。
4. 老板最有价值的视角不是看每个员工写了什么，而是看“哪一批、哪个车间、哪个环节、哪类异常正在影响产量/质量”。

因此飞书多维表格的设计目标应该是：

- 入口简单：员工/主任少填、少选、少写。
- 中间结构化：所有语音和文字最终落成标准字段。
- 出口聚焦：老板只看异常、趋势、负责人和处理状态。

## 8. 推荐系统架构

### 8.1 总体结构

可以把系统分成 5 层：

1. 员工/主任入口层
2. AI 理解与结构化层
3. 飞书多维表格数据层
4. 自动化与看板层
5. 老板问答与决策层

### 8.2 员工/主任入口层

第一版入口不要复杂。

可选入口：

- 飞书表单/多维表格表单视图；
- 飞书群里的机器人入口；
- 手机桌面快捷方式打开指定表单；
- 手机系统自带语音输入，把语音转成文字；
- 后续再做独立网页或小程序式“按住说话”入口。

入口设计原则：

- 员工只看到自己要填的 3-5 个按钮；
- 不让员工直接面对复杂数据库；
- 尽量用大按钮、少字段、默认值；
- 异常反馈入口要比正常记录入口更显眼。

示例入口：

- 今日晨会纪要
- 员工临时反馈
- 现场物料巡查
- 操作过程巡查
- 异常拍照上报

### 8.3 AI 理解与结构化层

AI 不直接“管理工厂”，只做翻译官和整理员。

AI 接到原始输入后做 5 件事：

1. 识别车间和表格类型。
2. 把自然语言拆成固定字段。
3. 用车间知识库把俗称翻译成标准名。
4. 判断是否产生异常事件。
5. 给出置信度，低置信度的交给人工确认。

示例：

原始输入：

> B 组 7-3 有一个病虫，旧杯盖今天还是有点盖不严。

结构化结果：

| 字段 | 结果 |
| --- | --- |
| 车间 | 浸染/制盖杯 |
| 批次/组别 | B 组 7-3 |
| 异常 1 | 病虫 1 个 |
| 异常 2 | 旧杯盖盖不严 |
| 影响环节 | 操作过程/物料 |
| 是否进入老板看板 | 是 |

### 8.4 飞书多维表格数据层

飞书多维表格相当于项目的业务数据库。

建议采用“主表 + 明细表 + 字典表 + 异常表”的结构，而不是把所有字段塞进一张大表。

核心表：

1. 车间表
2. 人员表
3. 批次表
4. 晨会纪要表
5. 员工反馈表
6. 巡查记录表
7. 巡查项目明细表
8. 物料/虫体/杯体字典表
9. 异常事件表
10. 处理跟进表

其中最重要的是“批次表”和“异常事件表”。

批次表解决追溯：

- 今天哪个车间；
- 哪一批虫/杯/料；
- 谁操作；
- 哪天巡查；
- 出现过什么异常；
- 最后合格率是多少。

异常事件表解决老板看板：

- 哪些没完成；
- 哪些不合格；
- 哪些需要处理；
- 谁负责；
- 是否复查完成。

### 8.5 自动化与看板层

飞书内部可以做很多自动化：

- 表单提交后自动生成记录；
- AI 识别到异常后自动写入异常事件表；
- 异常状态变更后提醒负责人；
- 每晚自动生成日报；
- 每周自动按车间统计异常类型；
- 合格率低于阈值自动标红；
- 连续多日重复异常自动升级。

老板看板不应该展示所有流水账，而是展示：

- 今日异常数量；
- 按车间分布；
- 按异常类型分布；
- 未处理异常；
- 重复出现异常；
- 合格率趋势；
- 产量/任务完成趋势；
- 需要老板拍板的问题。

### 8.6 老板问答与决策层

老板端可以分两种：

第一种：飞书看板。

适合固定问题：

- 今天有哪些异常？
- 哪个车间问题最多？
- 哪些异常还没处理？
- 今天合格率最低的是哪一项？

第二种：AI 问答。

适合临时问题：

- 最近一周空杯问题是不是变多了？
- 旧杯旧盖导致的问题集中在哪个车间？
- 哪个主任反馈问题最及时？
- 病虫异常和哪个批次关系最大？

AI 问答要注意：它应该只基于飞书里的结构化数据回答，不要凭空编。

## 9. 我们具体要做的事情

### 9.1 第一阶段：业务和表格梳理

目标：把纸表翻译成数据库。

要做：

1. 收集五个车间所有空白表格和已填样表。
2. 拆每张表的字段。
3. 标注哪些字段是人工输入、哪些字段可 AI 提取、哪些字段可自动计算。
4. 定义异常类型字典。
5. 定义老板最关心的指标。

交付物：

- 车间表格字段清单；
- 异常类型清单；
- 第一版飞书表结构设计。

### 9.2 第二阶段：飞书数据底座

目标：先不用复杂 AI，也让数据能进来、能统计。

要做：

1. 建飞书多维表格。
2. 建车间、人员、批次、晨会、巡查、异常等基础表。
3. 建表单入口。
4. 建基础视图：按日期、车间、异常状态、负责人查看。
5. 建老板看板初版。

交付物：

- 可用的飞书多维表格；
- 员工/主任录入入口；
- 老板异常看板。

### 9.3 第三阶段：AI 结构化

目标：减少主任手写和手填。

要做：

1. 为晨会纪要做 AI 字段拆解。
2. 为员工反馈做 AI 异常归类。
3. 为巡检记录做“默认合格 + 异常提取”。
4. 建车间知识库，把俗称映射成标准名。
5. 加人工确认流程，避免 AI 误填正式记录。

交付物：

- AI 整理提示词；
- 标准化输出 JSON；
- 异常自动入库流程；
- 人工确认机制。

### 9.4 第四阶段：老板问答

目标：老板可以问数据，而不是自己翻表。

要做：

1. 设计老板常问问题模板。
2. 从飞书多维表格读取相关记录。
3. AI 汇总回答，只引用真实数据。
4. 对高风险回答附上来源记录。

交付物：

- 老板问答入口；
- 常用问题按钮；
- 每日/每周自动总结。

### 9.5 第五阶段：语音入口升级

目标：从“手机自带语音输入”升级到“按住说话”。

要做：

1. 做一个轻量网页或飞书机器人入口。
2. 接语音转文字服务。
3. 把转写文字交给 AI 结构化。
4. 写入飞书多维表格。

这一阶段适合放到后面，因为它会增加接口费用、开发复杂度和维护成本。

## 10. 第一版最推荐架构

第一版建议采用：

> 飞书多维表格 + 表单入口 + 手机自带语音输入 + AI 文本结构化 + 异常看板

不建议一开始采用：

> 独立 App + 实时语音识别 + 全自动判断 + 全车间一次性上线

原因：

1. 工厂现场最怕复杂，复杂就没人填。
2. 第一版重点是跑通数据闭环。
3. AI 最容易出价值的是整理语言和汇总异常，不是替代全部人工判断。
4. 飞书本身已经能承担数据库、表单、权限、视图、看板、自动化的大部分工作。
